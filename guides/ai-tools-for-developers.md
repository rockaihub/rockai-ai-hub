# 开发者 AI 工具指南

AI 对开发者最实用的价值，是把重复性工作、陌生代码理解、错误排查和文档整理的成本降下来。它不能替代工程判断，但可以成为很好的结对助手。

## 开发者常用工具

| 工具 | 适合用途 |
| --- | --- |
| Cursor | 项目级代码理解、修改、重构、测试 |
| ChatGPT | 技术解释、方案讨论、代码片段、排错 |
| Claude | 长文档、架构说明、需求整理、代码评审辅助 |
| Gemini | 技术资料检索、多模态截图理解 |
| Perplexity | 查资料、找文档、追踪来源 |

## 适合 AI 辅助的开发任务

### 1. 阅读陌生项目

接手项目时，可以先让 AI 总结结构。

```text
请阅读这个项目，说明：
1. 项目入口在哪里
2. 核心模块有哪些
3. 数据流大致如何运转
4. 新增一个页面或接口通常应该改哪些文件
```

### 2. 解释复杂代码

不要只问“这段代码什么意思”，可以要求 AI 从输入、输出、边界和副作用解释。

```text
请解释这个函数：
1. 输入参数含义
2. 返回值含义
3. 主要流程
4. 可能的边界情况
5. 是否有潜在问题
```

### 3. 辅助实现小功能

AI 更适合边界清晰的任务。

```text
请为登录表单增加邮箱格式校验。
要求：
1. 保持现有 UI 风格
2. 错误提示显示在输入框下方
3. 不改动无关文件
4. 补充对应测试
```

### 4. 错误排查

把报错、触发步骤、相关文件和你已经尝试过的方法告诉 AI。

```text
这是运行测试时的报错。
请帮我判断最可能的原因，并给出排查顺序。
不要直接大范围重构，优先找最小修改方案。
```

### 5. 代码评审辅助

AI 可以帮你发现明显问题，但不能替代正式评审。

```text
请以代码评审的角度检查这次改动，重点关注：
1. 逻辑漏洞
2. 边界情况
3. 类型问题
4. 是否缺少测试
5. 是否引入不必要复杂度
```

## Cursor 使用建议

### 先读后改

让 Cursor 先理解项目模式，再做修改。这样更容易保持代码风格一致。

### 控制修改范围

给 AI 明确文件范围和验收标准，避免它修改过多无关内容。

### 每次改动后运行验证

至少运行一种检查：

- 单元测试
- 类型检查
- 编译
- lint
- 本地页面验证

### 保留人工审查

AI 可能生成看似合理但隐藏问题的代码。你需要检查：

- 是否符合业务需求
- 是否破坏旧逻辑
- 是否处理边界情况
- 是否泄露敏感信息
- 是否有足够测试

## 不适合完全交给 AI 的任务

- 核心架构决策
- 安全策略设计
- 复杂权限系统
- 数据库迁移方案
- 支付、财务、风控相关逻辑
- 大规模重构
- 生产事故处理

这些任务可以让 AI 辅助分析，但最终判断应由开发者或团队负责。

## 推荐开发工作流

1. 描述需求和边界
2. 让 AI 阅读相关代码
3. 要求 AI 给出修改计划
4. 分小步实施
5. 运行测试和类型检查
6. 人工审查 diff
7. 让 AI 辅助补充测试和文档

## 团队使用注意事项

团队引入 AI 编程工具前，建议明确：

- 哪些代码可以输入 AI 工具
- 是否允许上传内部文档
- 是否需要统一账号或服务支持
- 是否需要提示词和工作流规范
- 如何评审 AI 参与生成的代码
- 如何处理安全、隐私和合规要求
